ID Shield for Laserfiche
Large Scale Repository Configuration

  • Assemble the pre-created Large Scale Repository

    • Ensure Microsoft SQL (Express or full) is installed along with Microsoft SQL Management Studio
    • Set an unlimited timeout value for SQL server
      • Open Microsoft SQL Management Studio and connect to the local server.
      • Right-click on the local server in the Object Explorer pane and select Properties.
      • Select the Connections section.
      • Enter 0 in the Remote query timeout box.
      • Stop and re-start the Microsoft SQL service.
    • Ensure Laserfiche Client & Server is installed including the adminstrative modules in the Client install.
    • A 200,000 document database for test is avalaible here: I:\Common\Testing\Product Testing\Laserfiche\Large Scale Repository
    • To set it up:
      • Register the repository
        • Copy Large_Scale_Repository.zip to the machine that will be hosting the database and extract it to a directory to be used to store the repository (hereafter refered to as [Repository directory]).
        • Ensure Micorosoft SQL (Full or Express) is installed included Microsoft SQL Management Studio
        • In Microsoft SQL Management Studio, right-click on Databases and select "Attach.."
        • Attach the database [Repository directory]\database\Large_Scale_Repository.mdf
        • Open the Laserfiche Administration Console
        • If the local machine is not already registered, register it using New Server Registration...
        • Once the machine is registered, right-click the "Repositories" node and select "Register Repository"
        • Laserfiche Server should be the local machine name.
        • Password should be the associated password for the Laserfiche Server (try "a" if you didn't create the password)
        • DBMS should be Microsoft SQL
        • Press Next
        • Repository Name = "Large Scale Repository" (but could be anything you want)
        • Repository Path Specify the [Repository directory]
        • Location "Under the the Laserfiche files directory"
        • Press Next
        • Server should be the local machine name
        • Select Use Windows Authentication.
        • Database Select "Large_Scale_Repository"
        • Press Next
        • Click Next one more time to accept the settings

      • Set the volume path
        • Open the Laserfiche Administration Console
        • If the mahine you are woking on does not appear beneath the Laserfiche Administration Console node...
          • Right-click on the Laserfiche Administration Console node and select New Server Registration...
          • Enter the name of the machine you are working on.
        • Select Laserfiche Administration Console/[Machine name]/Repositories/[Large Scale Repository] and login as username: Admin, password: a.
        • Select the Volumes node.
        • Right-click on the DEFAULT volume and select Properties
        • On the Path tab change the Fixed Path to [Repository directory]\DEFAULT. (It is possible you may first have to rename the existing DEFAULT directory under [Repository directory], then copy it back over the DEFAULT directory that is created once you have updated the volume path).

      • Populate the repository
        • Open a Laserfiche Client (7.2 or higher) and login as user: Admin, password: a
        • Decide where you want the initial 200,000 document to reside. For example, create a folder called 01 beneath the Documents folder. Open that folder, the select File | Import... menu in the Client.
        • Click Options... in the lower right and ensure the following settings:
          • Unchecked: Always show Document Metadata dialog
          • Unchecked: OCR/Extract text
          • Select: (Save images as) one page documents
          • Unchecked: New text page every [x] lines
        • In the I:\Common\Testing\Product Testing\Laserfiche\LargeScaleDatabase directory select Large Scale Repository Part1.lfb and Large Scale Repository Part2.lfb
        • Press Import

      • Repository information
        • The repository has 200,000 documents with ~392,000 pages.
        • There a 3 existings users all with the password a:
          • Admin: Super-user with all rights and privileges
          • User: User with all document handling rights except to adjust access rights to documents and folder and but without repository privileges such as configuring metadata for the repository.
          • Peon: User with almost no rights except to search for and view documents.
        • Each of these users is beneath a corresponding user group: Administrators, Users and Peons. New users can easily be added matching the privileges of an existing user by adding them to the corresponding group.
        • Large Scale Repository Part1.lfb contains un-tagged documents.
        • Large Scale Repository Part2.lfb contains documents all tagged as ID Shield - Needs Processing and they will appear tagged as such once the repository is configured for use with ID Shield. This was not the original intention, but I don't think it substantially degrades the Endurance Test since the second half of the endurance test involves clearing the tags from all records in the repository and re-submitting from scratch.
