Goals of the "Feedback collection" feature:

  1. To know actual capture rate at client site from Time1 to Time2.
  2. Be able to diagnose why certain data was not captured, and be able to improve our product so that the data can be captured in the future.

High Level Requirements:

  1. The user shall be able to specify options for feedback collection via a UI.
    1. Whether feedback collection is on or off
    2. Option to turn off feedback collection when data has been collected for X documents
    3. Option to turn off feedback collection on a certain date 
    4. Folder to store feedback data
    5. Three options for:
      1. Don't collect source documents
      2. Collect source-document at packaging time
      3. Collect source-document at rule-execution time
    6. A checkbox to convert source document to text if source document collection is enabled
    7. Option to specify AttributeNames for which feedback data should be collected.
  2. Packaging feedback data and sending to UCLID
    1. The user shall be able to package the feedback data into one file which they can mail or FTP to UCLID.
    2. After packaging the feedback data, the user shall be able to decide whether the feedback data should be "cleared".
    3. The packaging utility shall optionally allow the user to select source documents to be packaged along with the rest of the feedback data.
    4. While packaging the feedback data, the utility shall maintain an error log of all errors encountered and the error log file shall be added to the package as well.
    5. The packaging utility shall record an entry in the error log if it encounters any rule-execution-records for which the corresponding .rsd file has a newer modify timestamp than the time of execution of the ruleset.
    6. The packaging utility shall package all files such that they can be unpacked with the exact timestamps that they had at the time of packaging.
    7. The packaging utility shall create and package two files called "RunRules.dat" and "CompareResults.dat" which can readily be used by UCLID's automated ruleset testing component for testing and report generation purposes.
    8. [New] The packaging utility shall also package a copy of any rulesets (rsd files) referenced in the feedback collection data being packaged.
    9. [New] All packaged files shall retain their last-modify-date file attribute when unpacked.
  3. Each execution of the RuleSet shall have its feedback data tagged with a unique ID specific to that execution.
  4. The following information shall be collected as part of the feedback data for each execution of the RuleSet:
    1. One of the following containing Information about the source document:
      1. Name of an image
      2. Name of an USS file and name of the image associated with the USS file
      3. Name of a text file
      4. Text in the form of string passed in to IAttributeFinderEngine
    2. The filename of the RuleSet that was executed
    3. Information about the data that FlexIndex was able to capture at the field at the time that the RuleSet was executed
      1. IIUnknownVector<IAttribute>
    4. Optional information about the actual data (i.e. "correct data") that the end-user application stored into the database after the user has had a chance to edit + verify the data that FlexIndex presented to them
      1. IIUnknownVector<IAttribute>
      2. This piece of the data is optional because it requires integration work with the end-user application, which may not always be done.  Some of the other colleced feedback data is not as valuable if this piece of information doesn't exist.
    5. Start date and time of the execution
    6. Duration of execution in wall-clock time elapsed.
    7. Details of any exception that was thrown to the application.  The exception shall be stored in a file called {ID}.uex in the feedback data collection folder where {ID} is the rule execution ID.
  5. TODO: UCLIDException.uex...
  6. Utilities shall be developed that allow UCLID developers to:
    1. Unpack the feedback data received from the client to:
      1. Allow reproduction of the execution results seen by the client.
      2. Allow developers to debug and improve rules
    2. Allow developers to generate reports from the unpacked feedback data (NOTE: in the beginning we may not want to give the report generation utility to our customers, but on the long run we may so that they can generate the reports on their own)
      1. Reports about capture rates
        1. [New] Percentages for each attribute and sub-attribute
      2. Reports about throughput
        1. [New] Avg time for execution of a ruleset
        2. [New] Avg time to OCR a document and avg time to OCR a page
      3. [New] Reports about approximate keystroke reduction
        1. [New] Options to generate report based on single data-entry or double-blind entry

Other technical notes:

  1. Create a "Clear Attribute Value" Output Handler, which allows clearing out of attribute values for one or more named attribute's, where the attribute name can contain sub-attribute names like "Grantor-Grantee.Person"
  2. Create an X-path like query tool that allows us to query the engine's result with a string like Grantor-Grantee@Type=Grantee/Person and returns back an IIUnknownVector<IAttribute> with matching tree nodes and sub-nodes if available.
  3. Update all current splitter objects with an additional method which will take the values of the split parts, and return an IAttribute with subattributes containing the specified values for the various split parts.  Example of such a method is CreatePersonAttribute(strTitle, strFirst, strMiddle, strLast, strSuffix, bAutoBuildParent).  The bAutoBuildParent argument is available only in those splitter objects where the parent value can be computed based upon the values of the sub-attributes.
  4. Add <CompareResults> tag to syntax of automated ruleset tester component which will allow direct comparision of a found-attributes .voa file to an expected-attributes .voa file without having to run any rules.  Syntax for line is "<CompareResults>;FoundAttributesVOA;ExpectedAttributesVOA"
  5. Add <CompareResultsFolder> tag to syntax of automated ruleset tester component - similar functionality as above, but at a folder level.  The algorithm will find "*.correct.voa" and compare those voa files to corresponding ".found.voa" files.  Syntax for line is "<CompareResultsFolder>;folder_name"
  6. The AttributeFinderEngine shall be enhanced so that it always creates a <RuleExecutionID> tag in the AFDocument whenever any type of rule execution is invoked.
  7. [New] Add IAFUtility.GetAttributesFromAutoFile(strFileName) which returns an IIUnknownVector<IAttribute> and can read attributes from either .voa or .eav files
  8. [New] Update automated test components to use IAFUtility.GetAttributesFromAutoFile() so that they will also support .voa files.

Ideas to implement in the future:

  1. UCLID writes a utility to automatically package feedback data and send to UCLID via FTP, showing progress bar, allowing partial uploads which can be resumed later, prompting for ftp username and password, etc.
  2. [New] Convert EAV files to VOA files on the network and eliminate support for EAV files.
