ID Shield Workflow

Setup and Verification

  • For every ID Shield project, we receive the following items from our customers:
    1. A set of sample images.
      • Our customers often need to redact millions of images.
      • Since it is impractical to analyze all of the customer's images, we ask the customer to provide us with a small, representative sampling of the images that they want to redact.
      • Generally, this sample set is approximately 10,000 pages.
      • If the customer provides us with a set of sample images that is larger than 10,000 pages, we will take a random subset of 10,000 pages from the provided sample set.
      • The sample images are stored at K:\Common\Engineering\Sample Files.
    2. Directions about which field or fields to redact.
      • For example, a customer may want to redact social security numbers and dates of birth.
    3. Details about the customer's workflow, including:
      • Whether the customer will be verifying images or using fully automated redaction.
      • The desired capture rates.
      • Other specific constraints.

  • After the sample images are received, they must be verified to create Expected Redactions .evoa files.
    • First, the sample images are OCRed.
    • Next, rules are run on the OCRed sample images in order to create output .voa files that will be used to pre-populated the verification window with data. This step is not necessary, but it does speed up the verification process.
      • Generally, the rules that are run on the OCRed images are rules that were previously completed for another, similar project.
      • If we have never completed a similar project before, we will run a "skeleton" rules set on the OCRed images.
        • These "skeleton" rules will only contain the ComponentData redaction rules for the desired fields.
    • Lastly, each image is verified by a human operator and an Expected Redaction .evoa file is created.

Rule Writing

  • Rather than writing rules from scratch, rule writers generally take an existing set of rules and add new rules to them.
    • If we have previously completed a project that is similar to the current project, the rules from the older project can be used as a basis for the current project.
    • If there is no suitable pre-existing rules set, you can start with a "skeleton" rules set that does nothing but call the appropriate attributes from ComponentData.
  • The focus of rule writing is different for customers who are verifying documents and customers who are automatically redacting documents.
    • For customers that are manually verifying documents, focus on increasing the number of data items found during verification while reducing the number of files selected for review.
      • Unless otherwise instructed, the percentage of items found during verification needs to be 99% or greater.
      • When writing redaction rules for customers who are verifying, write the rules in a manner that will increase the speed of verifiers.
        • Increase the automated capture rate and reduce the number of false positives as much as possible. The more items that the rules capture correctly, the less time verifiers have to spend adding redaction boxes or deleting incorrect redaction boxes.
        • Avoid over-redactions whenever possible. If a redaction box is too large, a verifier may have to spend time resizing it.
        • Avoid redacting the same item twice. Double redactions can be confusing to verifiers.
    • For customers that are automatically redacting documents, focus on increasing automated capture rate and reducing the number false positives.
      • Unless otherwise instructed, the automated capture rate should be over 95% and the ROCE should be 5 or greater.
      • Reduce the number over-redactions that cover non-sensitive data. Since the customer is not verifying, they will not have a chance to resize redactions that are too large.
      • Double redactions are acceptable for automated projects.
  • Use the Test Harness to test the improvements that you make to the rules.
    • You should record the results of your Test Harness tests at least once a day.
    • Every project has its own folder under I:\Common\Engineering\ProductDevelopment\AttributeFinder\Customer Specific Files. Record your Test Harness test results in the Performance Results spreadsheet that is located in the Performance Results subfolder of the project's folder.
  • Generally, you should continue to improve the rules either until you completely meet the requirements of the project or until you work an entire day an achieve less than a 2% improvement in capture rates.

Turning in a Project

  • When you think you are done with your rules, you will need to have another rule writer review them.
    • Explain how your rules work to the other rule writer.
    • The other rule writer will often catch mistakes or make suggestions about how you can improve your rules.
    • Be sure to correct and test any items that were discovered during rules review before you check in your rules.
  • After your rules have been reviewed, you will need to add your rules and testing files to Vault.
    • All redaction project have their own folder in Vault under $\Engineering\Rules\IDShield.
    • If the project folder does not exist, create it.
    • Add your rules to the Rules sub-directory of the project folder.
    • Add any .tcl and .dat files used for testing to the TestingFiles sub-directory of the project folder.
  • Next, you will need to add sample images to the delivery test.
    • The delivery test is a very short test that is run by the rules QA personnel just before the rules are sent to the customer to ensure that the rules are working properly.
    • The delivery test should generally only contain between 5 and 25 images.
    • In the project's folder under I:\Common\Engineering\ProductDevelopment\AttributeFinder\Customer Specific Files, there is a sub-folder labeled Delivery Test Files.
    • You will need to add your sample files to the Images sub-folder of the Delivery Test Files folder.
    • In general, you will need to include one example of every major document format for which you wrote rules.
    • Include the .uss files for any image that you add.
  • Be sure that you have recorded the final statistics for your rules in the Performance Results spreadsheet.
  • Notify the QA personnel that your rules are checked in and ready to be delivered.

Exercises

  1. Test Set #1 - Redact all social security numbers
    • Copy the sample images from I:\Common\Testing\Product Testing\IDShield_Training\Set001\Images to your local machine.
    • Create a rules file called Master.rsd that calls the ComponentData rules for social security numbers.
    • OCR the sample images and run Master.rsd on them.
    • Verify the sample images, saving the Expected Redactions files to an ExpectedRedactions subfolder.
    • Use Test Harness to compute initial capture rates of Master.rsd.
    • Improve rules to achieve >99% post-verification capture rate.

  2. Test Set #2 - Redact the first five digits of all social security numbers
    • Copy the sample images from I:\Common\Testing\Product Testing\IDShield_Training\Set002\Images to your local machine.
    • Create a rules file called Master.rsd that calls the ComponentData rules for social security numbers and chops off the last four digits of all numbers.
    • OCR the sample images and run Master.rsd on them.
    • Verify the sample images, saving the Expected Redactions files to an ExpectedRedactions subfolder.
    • Use Test Harness to compute initial capture rates of Master.rsd.
    • Improve the rules to achieve >99% post-verification capture rate.
