Pre-checkin tasks for developers and code-reviewers


  1. Is the new or modified code/algorithm correct and doing what it should?
  2. If new design has been implemented, or existing design has been refactored, has that been done in an appropriate way?
  3. Are the names of classes, variables, and methods appropriate?
  4. Is code in cpp files adequately documented?
  5. Are coding conventions being followed?
  6. Do method or function declarations in header/IDL/ODL files supported with purpose/promise/require comment blocks?
  7. Is the code in MFC UI handlers (e.g. OnClickedButtonX) wrapped within try/CATCH_AND_DISPLAY_ALL_EXCEPTIONS?
  8. Is the code within COM method implementations wrapped within try/CATCH_ALL_AND_RETURN_AS_COM_ERROR?
  9. Is the code in destructors wrapped within CATCH_AND_LOG_ALL_EXCEPTIONS?
  10. Is all allocated memory being released? Check regular heap allocations to pointers using the new keyword, as well as COM object allocations to smart pointers.
  11. Are pointers being checked with the ASSER_RESOURCE_ALLOCATION or other mechanism before they are dereferenced?
  12. Are exceptions coded to contain all possibly useful debug information available in a particular scope of code?
  13. Is the code thread-safe for a multi-processor/core machine?
  14. Is the actual visible-to-user-change at the application level working as expected?
  15. Have boundry conditions been checked?
  16. Have PVCS records been added and updated as appropriate for any new or modified functionality in the code about to be checked in?
  17. Has a full build of one or more related products been executed to ensure that code about to be checked in does not prevent compilation of code in other projects?
